description: The embedded dashboard — a React SPA served by NestJS that lists runs and renders each one as a graph across local and remote steps, with retry and cancel.
---

`@dudousxd/nestjs-durable-dashboard` mounts a control plane at `/durable`: a bundled React SPA
plus its JSON API. It reads straight from the [state store](/docs/stores), so it works without an
OpenTelemetry collector.

```ts
import { DurableDashboardModule } from '@dudousxd/nestjs-durable-dashboard';

@Module({
  imports: [DurableModule.forRoot({ store, transport }), DurableDashboardModule.forRoot()],
})
export class AppModule {}
```

`DurableModule` is global, so the dashboard resolves the engine and store automatically. Front the
base route with your own guard to protect it.

### Where it mounts

`forRoot()` defaults to `/durable` (SPA at `/durable`, JSON API at `/durable/api`). Pass `basePath`
to move it — e.g. under your app's `/api` prefix so the same auth/proxy rules cover the dashboard API:

```ts
DurableDashboardModule.forRoot({ basePath: '/api/durable' });
// SPA at /api/durable, API at /api/durable/api
```

The SPA's asset URLs and its API base are derived from `basePath` at serve time, so the bundle works
at any mount point. If your app sets a global prefix (`app.setGlobalPrefix('api')`), exclude the base
route so the path isn't prefixed twice — or fold it in by setting `basePath` to include the prefix.

### Serving it from an API pod (without processing there)

When you split your app into **API** and **worker** pods, you usually want the dashboard served from
the API (it handles HTTP) while only the workers process runs. Set `worker: false` on the API
instance's `DurableModule`:

```ts
DurableModule.forRoot({
  store,
  transport,
  worker: process.env.APP_TYPE !== 'API', // workers process; the API only serves the dashboard
});
```

With `worker: false` the engine is still fully available — the dashboard can **dispatch retries and
cancels**, and reads work — but the instance does **not** play the worker role: it won't register
`@DurableStep` handlers (so it never consumes the task queue), won't recover incomplete runs on boot,
and won't poll due timers. Leave that to the worker pods. So a **retry clicked in the UI on an API
pod still works**: the API dispatches the step, a worker picks it up and runs it. Only the processing
is gated, not the control plane. (Default is `worker: true`, so single-process apps need nothing.)

## What it shows

- **Runs** — every run with its status (running / suspended / completed / failed / cancelled),
  filterable.
- **Timeline as a graph** — the selected run rendered with `react-flow`: a node pipeline from
  start through each step to the terminal state, tagged by kind (local / remote / sleep / signal),
  with worker group and duration. This is the "see the whole flow" view — a workflow whose steps
  span apps shown as one rail.
- **Step timing** — click a step to inspect it. A remote step shows its **queued** time (how long
  it sat in the queue before a worker picked it up) next to its processing **duration**, so you can
  tell a slow handler apart from a backed-up worker pool. Queue-wait is reported by every transport
  and every language SDK (including the [Python worker](/docs/python)).
- **Actions** — retry a failed/suspended run, cancel a running one.

## API

The SPA is backed by a small JSON API you can also call directly:

| Method | Route | |
| --- | --- | --- |
| `GET` | `/durable/api/runs` | list runs (`?status=`, `?workflow=`) |
| `GET` | `/durable/api/runs/:id` | run + step timeline |
| `POST` | `/durable/api/runs/:id/retry` | resume the run |
| `POST` | `/durable/api/runs/:id/cancel` | cancel the run |
