`@dudousxd/nestjs-durable-transport-sqs` carries steps over **AWS SQS** ‚Äî the same role as the
[BullMQ transport](/docs/transports/bullmq), for teams already on SQS instead of Redis.

```bash
pnpm add @dudousxd/nestjs-durable-transport-sqs @aws-sdk/client-sqs
```

## How it works

Say a checkout workflow calls the `payments.charge-card` remote step. Its group is the name's first
segment ‚Äî **`payments`** ‚Äî so it runs on a worker in the `payments` group:

```ts title="the step + the call"
// chargeCard = remoteStep({ name: 'payments.charge-card', ‚Ä¶ })  ‚Üí group "payments"
const { chargeId } = await ctx.call(chargeCard, { orderId, amountCents });
```

With the default `durable` prefix, that one call moves three messages across two queues:

1. **Dispatch** ‚Üí the engine sends the `RemoteTask` JSON to the group's tasks queue,
   **`durable-tasks-payments`** (`<prefix>-tasks-<group>`). One queue per group, so a `payments`
   worker never sees `emails` tasks.
2. **Run** ‚Üí a worker in the `payments` group long-polls `durable-tasks-payments`, runs its
   `payments.charge-card` handler, then sends the `StepResult` JSON to the shared results queue,
   **`durable-results`** (`<prefix>-results`) ‚Äî every group replies on the same one.
3. **Resume** ‚Üí the engine long-polls `durable-results`, matches the result to the suspended
   `ctx.call` by `stepId`, checkpoints it, and the workflow continues.

```text
 engine ‚îÄ‚îÄRemoteTask‚îÄ‚îÄ‚ñ∂ durable-tasks-payments ‚îÄ‚îÄpoll‚îÄ‚îÄ‚ñ∂ payments worker
   ‚ñ≤                                                          ‚îÇ
   ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ poll ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ durable-results ‚óÄ‚îÄ‚îÄStepResult‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
```

```ts title="engine side"
// resolves durable-tasks-<group> + durable-results by name (auto-creates them if autoCreate)
new SqsTransport({ client: new SQSClient({ region: 'us-east-1' }), autoCreate: true });
```

```ts title="worker side ‚Äî the payments group"
const transport = new SqsTransport({ client, group: 'payments' });
transport.handle('payments.charge-card', async ({ orderId, amountCents }) => ({
  chargeId: await charge(orderId, amountCents),
}));
```

Add a second group and you get a second tasks queue automatically ‚Äî a `notifications` worker
(`group: 'notifications'`) polls **`durable-tasks-notifications`** and replies on the same
`durable-results`. Reusing queues you already provisioned? Skip name resolution by passing their
URLs directly:

```ts
new SqsTransport({
  client,
  group: 'payments',
  taskQueueUrl: () => 'https://sqs.us-east-1.amazonaws.com/1234567890/orders-charge-card',
  resultsQueueUrl: 'https://sqs.us-east-1.amazonaws.com/1234567890/orders-results',
});
```

SQS is at-least-once ‚Äî a task whose worker crashes before deleting it is redelivered after the
visibility timeout ‚Äî so keep handlers **idempotent**.

## Options

| Option | Description |
| --- | --- |
| `client` | An `@aws-sdk/client-sqs` `SQSClient`. |
| `group` | The worker group this instance serves ‚Äî required to register `handle()`s. |
| `prefix` | Namespaces the fallback `<prefix>-tasks-*` / `<prefix>-results` names. Default `durable`. |
| `autoCreate` | Create the by-name queues (`durable-tasks-<group>`, `durable-results`) if missing. Ignored for queues you supply via `taskQueueUrl` / `resultsQueueUrl` ‚Äî those must already exist. |
| `taskQueueUrl` / `resultsQueueUrl` | Reuse existing queues instead of resolving by name (a literal URL or a resolver). |

The message body is the documented `RemoteTask` / `StepResult` JSON ‚Äî identical to BullMQ ‚Äî so a
[Python worker](/docs/python#aws-sqs) on the same queues interoperates.
